1. PROJECT OVERVIEW
================================================================================

Project Name: The White Laser
Type: E-commerce Platform & Vector Editor (B2C/B2B)
Purpose: Premium product configurator and storefront for slate and wood products
         with integrated vector editor for laser engraving design customization

Key Capabilities:
  • Product catalog browsing and filtering
  • Interactive vector editor (Fabric.js-based canvas)
  • User authentication with email verification
  • Design customization and saving
  • Order management system

3. DATABASE ARCHITECTURE
================================================================================

Database System: PostgreSQL (Neon Serverless)
ORM: Prisma 7 with @prisma/adapter-pg

DATABASE MODELS & RELATIONSHIPS:

A. USER (Authentication & Profile)
   ├─ id (cuid - unique identifier)
   ├─ name (optional - display name)
   ├─ email (unique - login identifier)
   ├─ emailVerified (DateTime - marks verified accounts)
   ├─ password (hashed with bcrypt)
   ├─ image (optional - profile picture)
   ├─ role (USER or ADMIN)
   ├─ createdAt & updatedAt (timestamps)
   └─ Relations:
      ├─ designs[] (1:N - User can have many designs)
      └─ orders[] (1:N - User can have many orders)

B. DESIGN (Canvas Designs)
   ├─ id (cuid)
   ├─ userId (foreign key - required, cascade delete)
   ├─ name (default: "Design fără titlu")
   ├─ canvasJson (String - full Fabric.js canvas.toJSON() payload)
   ├─ createdAt & updatedAt
   └─ Relations:
      └─ user (N:1 - belongs to User)

C. PRODUCT (Catalog Items)
   ├─ id (cuid)
   ├─ name (product name)
   ├─ description (optional - product details)
   ├─ price (Float - unit price in RON)
   ├─ imageUrl (optional - product image)
   ├─ category (default: "General")
   ├─ isCustomizable (boolean - can be used with custom designs)
   ├─ createdAt & updatedAt
   └─ Relations:
      ├─ uploadedDesigns[] (1:N - uploaded template designs)
      └─ orderItems[] (1:N - referenced in orders)

D. VERIFICATION TOKEN (Email Verification)
   ├─ id (cuid)
   ├─ email (recipient email)
   ├─ token (unique - verification code)
   └─ expires (DateTime - token expiration)

E. UPLOADED DESIGN (Product Templates)
   ├─ id (cuid)
   ├─ productId (foreign key - required, cascade delete)
   ├─ fileUrl (path to uploaded design file)
   ├─ fileName (original filename)
   ├─ createdAt
   └─ Relations:
      └─ product (N:1 - belongs to Product)

F. ORDER (Purchase Orders)
   ├─ id (cuid)
   ├─ userId (foreign key - optional, set null on user delete)
   ├─ status (PENDING, PROCESSING, SHIPPED, DELIVERED, CANCELLED)
   ├─ total (Float - order total in RON)
   ├─ createdAt & updatedAt
   └─ Relations:
      ├─ user (N:1 - belongs to User, optional)
      └─ items[] (1:N - order line items)

G. ORDER ITEM (Order Line Items)
   ├─ id (cuid)
   ├─ orderId (foreign key - required, cascade delete)
   ├─ productId (foreign key - required)
   ├─ productName (String - snapshot of name at purchase)
   ├─ price (Float - snapshot of price at purchase)
   ├─ quantity (Int)
   └─ Relations:
      ├─ order (N:1 - belongs to Order)
      ├─ product (N:1 - belongs to Product)
      └─ customDesign (1:1 optional - OrderCustomDesign)

H. ORDER CUSTOM DESIGN (Custom Design Files in Orders)
   ├─ id (cuid)
   ├─ orderItemId (unique foreign key - 1:1 relationship)
   ├─ fileUrl (path to custom design file)
   ├─ fileName (design filename)
   ├─ createdAt
   └─ Relations:
      └─ orderItem (1:1 - belongs to OrderItem)


KEY DATABASE PATTERNS:
  • cuid() for all IDs (cryptographically secure, sortable)
  • Soft relationships with snapshots (orderItem stores productName & price at purchase time)
  • Cascade deletes for user/order/product deletions
  • Set null for optional user references (guest orders possible)

6. AUTHENTICATION FLOW (NextAuth + Email Verification)
================================================================================

REGISTRATION FLOW:
  1. User fills registration form (email, password, name)
  2. POST /api/auth/register
  3. Password hashed with bcryptjs
  4. User created in database (emailVerified = null)
  5. VerificationToken generated with unique token
  6. Email sent via Nodemailer with verification link
  7. User redirected to login page
  
VERIFICATION FLOW:
  1. User clicks email verification link
  2. POST /api/auth/verify with token
  3. Token validated against VerificationToken
  4. If valid: User.emailVerified = now(), token deleted
  5. User can now login

LOGIN FLOW:
  1. User submits email and password on /auth/login
  2. NextAuth CredentialsProvider validates credentials
  3. Email must be verified (emailVerified is not null)
  4. JWT created and stored in session cookie
  5. Session includes: id, email, name, role
  6. Redirect to dashboard/home

SESSION MANAGEMENT:
  • JWT-based sessions (cookies)
  • Session callbacks extend JWT with id and role
  • Middleware protects routes requiring authentication
  • useSession() hook for client-side auth checks

9. VECTOR EDITOR ARCHITECTURE (Fabric.js)
================================================================================

The vector editor is built with custom React hooks that isolate Fabric.js logic
from UI components. Page components call hook APIs only - no raw fabric.* calls.

CUSTOM HOOKS (Canvas Functionality):

useFabricCanvas
  • Initializes Fabric.js canvas on mount
  • Sets up ResizeObserver for responsive sizing
  • Calculates initial zoom to fit 4000mm document width
  • Saves canvas reference to editorStore
  • Handles canvas cleanup on unmount

useEditorTools
  • Manages tool activation (select, rect, ellipse, line, text, pan, draw)
  • Creates objects based on active tool
  • Handles tool-specific event management

useCanvasEvents
  • Registers Fabric.js event listeners (object:added, object:modified, etc.)
  • Syncs canvas state to Zustand store
  • Updates selection when objects are modified

useSelectionSync
  • Keeps editorStore.selectedObjects synchronized with canvas selection
  • Triggers UI updates when selection changes

useHistory (Undo/Redo)
  • Records canvas state snapshots in editorStore
  • Implements undo() and redo()
  • Prevents recursive snapshots with isRestoringHistory flag
  • Limits history to 50 snapshots

useAlignment
  • Align selected objects (left, center, right, top, middle, bottom)
  • Distribute objects evenly

useGrouping
  • Group selected objects together
  • Ungroup grouped objects

useCornerRadius
  • Apply rounded corners to rectangle objects

useSnapping
  • Display snap guides when moving/resizing objects
  • Snap to grid, guides, or object edges

useDrawingMode
  • Toggle freehand drawing mode
  • Manage brush properties

useCanvasNavigation
  • Pan canvas by dragging
  • Zoom in/out
  • Keyboard navigation (arrow keys)
  • Zoom to fit all objects

useExport
  • Export canvas to PNG format
  • Export canvas to SVG format

useSave
  • Save design JSON to database
  • Call /api/designs endpoint
  • Store canvas.toJSON() as canvasJson

useProductTemplate
  • Load product boundary/template onto canvas
  • Initialize canvas with product shape

useObjectUpdater
  • Batch-update properties on selected objects
  • Update fill color, stroke, opacity, etc.

COORDINATE SYSTEM:
  • Document width: 4000mm (internal coordinate space)
  • Initial zoom calculated to fit document in viewport
  • Rendering pixels converted from millimeters

CANVAS COMPONENTS:

EditorCanvas.tsx
  • Wrapper component that calls all hooks
  • Renders <canvas> element with fabric attached
  • Handles responsiveness via ResizeObserver

EditorLayout.tsx
  • Overall editor page layout
  • Canvas on left, properties/tools on right
  • Toolbar at top

PropertiesBar.tsx
  • Properties panel for selected objects
  • Update object properties in real-time

AlignDistributePanel.tsx
  • Alignment and distribution UI

svg-canvas.tsx
  • Alternative SVG rendering of canvas

18. SUMMARY & KEY TAKEAWAYS
================================================================================

The White Laser is a modern, full-stack e-commerce platform with an integrated
vector editor, built with:

FRONTEND:
  • Next.js 16 (App Router) + React 19 + TypeScript
  • Tailwind CSS + shadcn/ui for consistent UI
  • Custom Fabric.js hooks for canvas editing
  • Zustand for state management

BACKEND:
  • NextAuth for authentication
  • Prisma 7 ORM with PostgreSQL database
  • API routes for backend services
  • UploadThing for file management

DATABASE:
  • 8 models: User, Design, Product, Order, OrderItem, OrderCustomDesign,
    VerificationToken, UploadedDesign
  • Relationships designed for e-commerce and design management
  • Snapshots stored for orders (productName, price at purchase time)

KEY ARCHITECTURAL PATTERNS:
  • Hook-based canvas functionality (separation of concerns)
  • Zustand stores for persistent state
  • API routes for backend operations
  • shadcn/ui for UI consistency
  • TypeScript strict mode for type safety
  • Middleware for route protection

The project follows modern React patterns, maintains clear separation of concerns,
and provides a solid foundation for continued development and scaling.
